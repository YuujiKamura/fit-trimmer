# FIT TRIMMER (フィットトリマー) 🚴⚡🎥

## ❓ これは何？ (What is this?)

サイクリング動画に走行データ（GPSログ）をメーター風に重ね合わせて表示するHUD動画などを作成する際、**「動画と走行ログのスタート位置を同期してトリミングする作業」**を自動化・効率化するツールです。

*   **デスクトップアプリ版 (Windows / HUDエンコーダー搭載) [メイン]**: [Releases](https://github.com/YuujiKamura/fit-trimmer/releases) ページからビルド済みバイナリをダウンロードして使用できます。スピードやパワー、心拍数などのHUDメーター情報を重ね合わせた動画を直接エンコード出力できます。
*   **Webアプリ版 (トリミングのみ / GitHub Pages)**: [https://yuujikamura.github.io/fit-trimmer/](https://yuujikamura.github.io/fit-trimmer/)
*   **100% ローカル処理**: ファイルをサーバーにアップロードしないため、個人情報の保護はもちろん、巨大な動画ファイルでも通信データ量を消費せず超高速で処理できます。

---

## 🖥️ デスクトップアプリ版 & HUD エンコーダー (Desktop App & HUD Encoder)

動画上にスピード、ケイデンス、心拍数、パワー、斜度、獲得標高グラフなどのメーター情報（HUD: Heads-Up Display）を直接オーバーレイして動画を合成・出力できます。

### 🌟 主な特徴
- **動画分割・順次エンコード**: タイムライン上に複数の分割ポイントを設定し、動画を分割して順次エンコードできます。各パートの完了後に一時ファイルを自動クリーンアップするため、Cドライブの容量（テンポラリ領域）を大幅に節約できます。
- **高速ローカルエンコード**: H.264 ハードウェアアクセラレーション（Intel QSV, NVIDIA NVENC, AMD AMF）を自動検出して利用し、高速（実時間の1.5x〜4x倍速）にHUD動画を出力できます。
- **Google Drive (仮想マウント) 対応**: クラウドストレージ上の巨大ファイルを処理する際、必要なトリミング区間だけを自動でローカルキャッシュし、ネットワーク遅延やI/O制限を回避します。
- **プレビュータイムライン**: GUI上で簡単に動画と走行データのズレ（タイムアライメント）を同期調整したり、トリミング範囲のドラッグ操作がスムーズに行えます。
- **安全な上書き防止**: 同名ファイルがある場合は上書き確認を行い、上書きか連番付き別名保存（または一時保存）かを選択できます。

### 🚀 使い方
1. [Releases](https://github.com/YuujiKamura/fit-trimmer/releases) からビルド済みバイナリをダウンロードしてインストール（またはリポジトリをクローンして `./gradlew :composeApp:run` で実行）。
2. 動画ファイルと `.fit` ファイルをドロップし、トリミング範囲を設定します。
3. HUDのデザインを微調整し、「HUD動画を出力」を実行します。

## 🛠️ トリミングの仕組み (How it Works)

本ツールは**時刻（UTC: 協定世界時）**をキーにして、動画と走行データを同期しています。

1.  **動画メタデータの解析**: 動画ファイル（`moov -> mvhd`）の撮影開始日時（UTC）と録画時間をミリ秒単位で抽出します。
2.  **FITデータのスライス**: 動画の撮影時間範囲に合致するレコードだけを切り出します。
3.  **走行データの自動補正**:
    *   **距離のリスタート**: 開始地点での累積距離を `0` メートルとし、以降の距離データから初期オフセットを減算します。
    *   **概要サマリーの書き換え**: セッションやラップの開始/終了時刻、走行時間、合計獲得距離などのサマリー情報を再計算して更新します。
    *   **CRCとサイズの再計算**: データの整合性を完全に修復し、Garmin ConnectやStrava等での「ファイル破損エラー」を防ぎます。

## 🎥 MP4 と LRV の関係性 (MP4 vs LRV)

GoPro等で撮影した本編動画（`.mp4`）とプレビュー用動画（`.lrv`）は、作成日時や録画秒数が完全に同一です。

*   **`.lrv` の使用を強く推奨**:
    本編の `mp4` はファイルサイズが非常に大きく処理が重くなりますが、 `lrv` は軽量です。トリミング処理や位置合わせ時のプレビュー表示のメモリ負荷を抑えて高速に処理できるため、 `lrv` をドロップして処理するのが最適です。

## ⏰ カメラの内蔵時計のズレと同期の問題

カメラの内蔵時計がズレていると、同期のタイミングもズレてしまいます。

*   **GPSデバイス（FIT側）**: GPS衛星から時刻を受信するため一般的に非常に高精度ですが、測位環境や記録開始時のタイミング等により数秒程度のズレが生じることがあります。
*   **カメラ側**: 手動設定された内蔵時計に依存するため、徐々にズレていきます。
*   **予防策（裏技）**:
    撮影を開始する前に、カメラをメーカーのスマホアプリ（GoPro Quik等）に一度接続すると、スマホの正確な時刻に自動同期され、ズレを防ぐことができます。

## 💡 【アイデア・未実装】緯度経度（GPS軌跡）を使った自動時間ズレ補正の検討（案）

カメラの時計がズレていても、位置情報を使って自動補正する技術的なアプローチ案です。**（※本ツールには実装されていません）**

### 1. 軌跡パターンマッチングによる自動同期の概念
動画側にも毎秒のGPS軌跡が記録されている場合、動画側のGPS軌跡をFITファイル側の軌跡の上に重ね合わせ、ズレが最も小さくなる時間差（タイムオフセット）を探索することで、自動同期を実現するアイデアです。

### 2. 録画開始時の数秒間（点）での簡易補正案
動画全体のGPSを記録し続けるとカメラのバッテリー負荷が高くなりますが、「録画開始時の数十秒間のGPS位置・速度・移動方向」をFITデータと照合（マップマッチング）することで、正確なGPS開始時刻を特定し、同期を完了させるアプローチです。
