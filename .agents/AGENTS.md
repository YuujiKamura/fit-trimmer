# Agent Rules

## Git Commits
- すべての実装、バグ修正、調整が完了した際には、**必ず変更されたファイルを git commit すること。**
- 作業の節目やターンの終わりには、コミット漏れがないかを常に `git status` で確認すること。
- コミットメッセージには、変更されたモジュールや修正内容を明瞭に記述すること。

## Versioning Rules (Git Tags)
バグ修正や機能追加を行いリリースを更新する際は、以下のセマンティックバージョニング（SemVer）ルールを厳格に適用すること：
- **PATCH（1.3.xの末尾）**: 細かなバグ修正、UIの崩れ調整、パフォーマンスチューニング、内部リファクタリングなど、「変なところの修正」のたびに必ずインクリメントする（例：`v1.3.0` -> `v1.3.1`）。
- **MINOR（1.x.0の真ん中）**: 新機能の追加、新しい設定項目の追加、大きなUI仕様の変更など、アプリに「非破壊的な機能・仕様変更」が入った場合にのみインクリメントする（例：`v1.3.0` -> `v1.4.0`）。
- **MAJOR（x.0.0の先頭）**: アプリケーション全体の設計刷新や、後方互換性のない破壊的変更が発生した場合にのみインクリメントする（例：`v1.3.0` -> `v2.0.0`）。
- リリース更新時は、必ず `git tag -a vX.Y.Z -m "..."` で注釈付きタグを作成し、`git push origin main --tags` でリモートにプッシュすること。
