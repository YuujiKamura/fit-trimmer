# Agent Rules

## Git Commits
- すべての実装、バグ修正、調整が完了した際には、**必ず変更されたファイルを git commit すること。**
- 作業の節目やターンの終わりには、コミット漏れがないかを常に `git status` で確認すること。
- コミットメッセージには、変更されたモジュールや修正内容を明瞭に記述すること。

## Versioning Rules (Git Tags)
バグ修正や機能追加を行いリリースを更新する際は、以下のセマンティックバージョニング（SemVer）ルールを厳格に適用すること。**ただし、開発環境での局所的な動作調整、デバッグコードの追加、軽微なバグ修正といった「些細な変更（つまらん変更）」のたびに毎回新規タグ（リリース）を作成してプッシュしてはならない。** リリースすべき重要なマージや主要なアップデートのタイミングにのみタグを作成すること。
**【警告】リリース（git tagの作成およびリモートへのプッシュ）は、エージェントが自己判断で勝手に行ってはならない。必ず実行前にユーザーに確認し、明示的な許可を得ること。**
- **PATCH（1.3.xの末尾）**: 細かなバグ修正、UIの崩れ調整、パフォーマンスチューニング、内部リファクタリングなど、「変なところの修正」のたびに必ずインクリメントする（例：`v1.3.0` -> `v1.3.1`）。
- **MINOR（1.x.0の真ん中）**: 新機能の追加、新しい設定項目の追加、大きなUI仕様の変更など、アプリに「非破壊的な機能・仕様変更」が入った場合にのみインクリメントする（例：`v1.3.0` -> `v1.4.0`）。
- **MAJOR（x.0.0の先頭）**: アプリケーション全体の設計刷新や、後方互換性のない破壊的変更が発生した場合にのみインクリメントする（例：`v1.3.0` -> `v2.0.0`）。
- リリース更新時は、必ず `git tag -a vX.Y.Z -m "..."` で注釈付きタグを作成し、`git push origin main --tags` でリモートにプッシュすること。

## Test Verification before Claiming Completion (成果の証跡提示)
- すべての実装、バグ修正、調整が完了し、完了報告（あるいはタグ bumps）を行う際には、単に「修正しました」と報告するのではなく、**必ず該当部分のテストを実行し、そのテストが正常に通過したことを示す出力ログ（テストの実行結果エビデンス）を最終報告に提示すること。**
- テストがない場合はテストを追加するか、手動での動作検証結果（コマンド実行エビデンス）を提示すること。

## Strict Test-Driven Development (TDD) Requirement (通らないテストの先行記述)
- 新機能の追加、バグ修正、仕様変更のいずれにおいても、**実装コードを書く前に、必ずそれを失敗させるための「通らないテスト（Red）」を最初に書くこと。**
- テストケースには「通って当たり前」の都合の良いダミーデータではなく、解像度の不一致や非同期遅延など、現実の運用時に発生しうるエッジケースを想定した「壊すためのテスト（実用的な不整合インプット）」を先行して記述すること。
- テストが意図した理由で正常に失敗する（Verify Red）ことを確認した後にのみ、それをパスさせる最小限の実装（Green）を記述すること。合格確認（Green）後にのみ、リファクタリングを許可する。

## Uncommitted Changes Safeguard (作業開始前の状態確認と未コミット資産の保護)
- **【必須】セッション開始直後の git status / git diff 実行:** セッションが開始された最初のターンにおいて、**必ず最初に `git status` と `git diff` を実行して、ワークスペース内に未コミットの変更（他のエージェントやユーザーによる書きかけのコードなど）が存在しないかを完全に把握すること。**
- **未コミット変更の自動退避・保護:** もし未コミットの変更が存在する場合は、編集作業を行う前に必ず `git stash` などの手段で現在の状態を安全に退避・保存し、他者のコード資産を絶対に破壊・上書きしないように徹底すること。
- **自己判断での git checkout / git reset の絶対禁止:** コンパイルエラーや競合が発生した場合であっても、`git checkout <file>` や `git restore <file>`、`git reset --hard` などの**不可逆なファイルリセット操作を自己判断で実行することは厳重に禁止する。** 変更を破棄・リセットする前には、必ず対象ファイルのリストと理由を提示し、ユーザーから明示的な許可（Yes/No）を得ること。
