# Lossless Refactoring (ロスレス・リファクタリング)

長時間のチャットセッションが続き、コンテキスト（アテンション）が肥大化した状態で大規模なコードの切り出しやリファクタリングをAIに実行させると、AIのアテンションが散漫になり、ハルシネーション（勝手なコード省略、インデント崩れ、括弧の閉じ忘れ）によって構文エラーの泥沼（応答の破綻）に陥る確率が極めて高くなります。

このスキルは、AI自身が自身の推論限界を自覚し、OSの決定論的パワーと Git サンドボックスを駆使して、安全かつ 100% 正確にリファクタリングを完遂するための手順を定義します。

---

## ワークフローの3大原則

1. **コードを出力させない（OS/シェルへの責任分離）**
   AIに数千行のコードをチャット上で生成させてファイルに上書きさせてはいけません。
   代わりに、対象ファイルの「何行目から何行目を切り出すか」の境界インデックスを特定し、PowerShell 等の OS コマンドを用いて物理的にファイルをスライス・結合します。

2. **失敗のメタデータ（落とし穴）を引き継いでリセットする**
   スライス範囲がズレてコンパイルエラーが発生したり、泥沼化の兆候（コンテキストの逼迫）が見られた場合は、迷わず `git restore` でワーキングツリーをクリーンに戻します。
   その上で、**「なぜ失敗したか（エラーログ、ズレたインデックス、文字コードの競合）」の引き継ぎメモ（ハンドオーバー）**をファイルに書き出し、セッションを一度リセット（初期化）して真っ新なコンテキストで引き継ぎメモから再開します。

3. **環境依存の罠（改行コード・エンコーディング）の絶対排除**
   PowerShell でファイルを処理する際は、必ず .NET の `[System.IO.File]` API 等を使用し、**BOMなし UTF-8** での読み書きを明示します。パイプ（`>` や `Out-File`）による暗黙の文字コード変換（SJISやUTF-16への化け）や、改行コード（CRLF/LF）の違いによるインデックスの1行のズレを完全に防ぎます。

---

## 詳細プロセスと実装手順

### STEP 1: 境界（スライスインデックス）の確定
まずは元のクリーンな状態で、切り出すべきクラスや UI ブロックの正確な開始行と終了行（および対応する 0-based インデックス）を特定します。
* `git show HEAD:path/to/file` 等を用いて、行の文字列を直接表示し、インデックスがずれていないか物理的に確認します。
* 空行や改行コードの影響でインデックスが 1つズレることが多いため、境界線の直上・直下の 2〜3 行を必ず事前に出力して照合します。

### STEP 2: OS/シェルによる物理スライスとマージ
PowerShell 内で、.NET API を用いてファイルを配列としてロードし、行スライスを安全に結合してファイルに書き込みます。

#### ⚠️ PowerShell の結合時の重要注意（配列結合の維持）
PowerShell で単一の文字列（定数）と文字列 of 配列（スライス）を単純に `+` で結合すると、**改行コードがすべて消滅し、1行の巨大な文字列に無理やり連結される** バグが発生します。
これを防ぐため、すべての単一の文字列を `@(...)` で配列化してから結合します。

```powershell
# クリーンに全行を読み込み
$utf8 = [System.Text.Encoding]::UTF8
$l = [System.IO.File]::ReadAllLines($mainPath, $utf8)

# 1. 結合パーツ of 定義 (単一文字列は必ず @() で配列化する)
$part1 = $l[0..182]                             # インポートや main 関数
$composeWindowDef = @('    var composeWindow: java.awt.Window? by remember { mutableStateOf(null) }')
$wHeader = $l[965..980]                         # Windowの宣言開始
$wCloser = @('    }', '}')                     # 閉じ括弧

# 2. 配列として安全に結合 (改行が完全に維持される)
$newMain = $part1 + $composeWindowDef + $wHeader + $wCloser

# 3. BOMなし UTF-8 でロスレス書き込み
[System.IO.File]::WriteAllLines($mainPath, $newMain, $utf8)
```

### STEP 3: 破綻時のハンドオーバー・リセット（引き継ぎ）
もしコンパイルエラーが解消されず、セッションの対話履歴が長くなりすぎた場合は、以下の手順を踏みます。

1. `git restore <file>` または `git checkout` で、ファイルをクリーンな状態に一度戻します。
2.  artifacts などの作業ディレクトリに `handover_notes.md` を作成し、以下を書き出します。
   - これまでのスライス範囲で何行目に何が含まれていたか（スライス設計図）
   - なぜ失敗したか（コンパイルエラーのログ、未定義になった変数・定数など）
   - リセット後の次セッションで実行すべき PowerShell コマンド
3. ユーザーにリセット（セッション初期化）を促し、クリーンなコンテキストで `handover_notes.md` を読み込ませて一瞬で完了させます。
